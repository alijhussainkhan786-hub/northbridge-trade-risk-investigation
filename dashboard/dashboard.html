Methodology & Limitations

What this dashboard is

A review-prioritisation tool built on a 12-stage fictional investigation methodology: brief → data model → synthetic data generation → data audit → SQL investigation → Python behavioural analysis → entity resolution/network analysis → explainable prioritisation framework → case files → claim-control gate → this dashboard.

What this dashboard is not

  • Not a determination-of-wrongdoing tool. No entity anywhere in this project is described as having committed an offense.
  • Not a black-box score. Every priority band traces to a named, capped, documented indicator (see Stage 7 indicator dictionary).
  • Not based on real entities. All data is synthetic, generated for this fictional exercise with a documented planted-pattern methodology, validated separately from the analyst-facing dataset.

QA Report — Pre-Publication Checks

Remaining Limitations

  • Case C (value/weight set) cannot be closed without external product-specialist input — outside this dataset's scope.
  • Cases A/B cannot be confirmed or refuted as to cause without external corporate-registry data — outside this dataset's scope by design.
  • Declared-relationship provenance (regulatory_filing vs. self-reported) is not differentially weighted in the current scoring framework — a known gap.
  • Small-n sub-strata (Corporate segment, several corridor cells) limit confidence at that granularity throughout the project.
  • This is a fictional, synthetic dataset. No finding should be read as a statement about any real entity, individual, or organisation.
